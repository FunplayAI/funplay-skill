# Data-Driven Content Pipeline

Use this reference when levels, shop items, rewards, chapters, difficulty, or tuning values need to move out of code and into data.

## Pattern

```text
Designer table or generated JSON
  -> Editor importer
  -> ScriptableObject assets or Addressable-style data
  -> runtime loader
  -> validation report
```

For an early build, the source data can be generated by scripts. Keep the same import boundary so designers can replace generated data later.

## Content Types To Data-Drive

- Levels, boards, waves, seeds, layouts, and difficulty buckets.
- Reward tables, currency sources/sinks, costs, and drop rates.
- Shop products, bundles, price tiers, labels, and bonus grants.
- Tutorial steps, chapter/meta unlocks, quests, achievements, and feature gates.
- Audio/music mapping, VFX mapping, and UI asset variants when they are tuned by content.

## Importer Shape

Create editor-only importers that:

1. Read CSV, JSON, or another structured source.
2. Validate required columns, ids, references, and numeric ranges.
3. Generate ScriptableObject assets in deterministic paths.
4. Delete or mark stale generated assets.
5. Save assets and print a concise summary.

Prefer structured parsers. Do not parse production CSV or JSON with fragile string splitting.

## Runtime Shape

- Runtime systems read generated assets, not raw design tables.
- Lookups should fail loudly in development and fall back safely in production when possible.
- Use stable ids for all content references.
- Add validation commands that can run without opening gameplay UI.

## Funplay MCP Verification

With Unity MCP available:

- Write importer and data classes as persistent `.cs` files.
- Recompile and inspect console errors.
- Run the importer through a menu item or editor code.
- Read back asset counts and sample fields.
- Enter Play Mode and confirm runtime loader resolves a known content id.

Do not claim a content pipeline exists until the import step and runtime lookup have both been verified.
